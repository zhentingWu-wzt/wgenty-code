# Scoped Viewer Recovery Design

## Problem

The TUI polls capability-scoped agent views to populate the main-window
subagent selector. `DaemonClient` starts with no viewer token, but production
startup never calls `create_viewer()`. The first scoped request therefore lacks
`X-Wgenty-Viewer-Token`, receives `404 Not Found`, and causes the progress
poller to stop permanently. No `AgentLocalView` event reaches the TUI, so
`SubagentTree::active_count()` remains zero and the selector is hidden.

Viewer tokens are daemon-local state. Even after startup initialization is
added, a daemon restart invalidates the cached token and recreates the same
failure unless scoped requests can recover.

## Goals

- Obtain a viewer token automatically before the first scoped agent request.
- Refresh a stale viewer token after an authentication-shaped `401` or `404`
  response and retry the scoped request once.
- Prevent a single transient polling failure from permanently disabling the
  subagent selector.
- Preserve strict capability scoping and the daemon's indistinguishable
  `Not Found` denial behavior.

## Non-goals

- Changing selector layout, keyboard navigation, or focus-view rendering.
- Expanding an agent's visibility beyond self and direct children.
- Adding unbounded HTTP retries or changing general daemon authentication.

## Design

### Viewer lifecycle

`DaemonClient` remains the owner of the shared viewer token. A private helper
ensures a token exists before a scoped request: it reuses the cached value when
present and otherwise calls the existing viewer-creation endpoint.

Scoped request methods send the cached token. If the daemon returns `401` or
`404`, the client clears/replaces the token by creating a fresh viewer and
retries the original request exactly once. A second failure is returned to the
caller. Other status codes are returned without refreshing because they do not
indicate an invalid viewer identity.

The retry is bounded to one refresh per request, preventing infinite retry
loops when the daemon or endpoint is unhealthy.

### Poller behavior

The sequential and parallel task/delegate pollers continue polling after an
individual `get_root_agent_view` error instead of terminating permanently.
Their existing 500 ms interval and maximum duration bound the retry rate and
lifetime. Errors are logged at debug or warning level with actionable context;
the next iteration may recover after daemon readiness or token recreation.

### Concurrency

Cloned `DaemonClient` values share the token through `Arc<RwLock<Option<String>>>`.
Token creation and refresh must avoid holding the write lock across HTTP
requests. Concurrent first requests may create more than one valid viewer;
the last stored token wins, which is safe because viewer creation is idempotent
from the client's perspective and viewers carry identical scoped privileges.

## Error handling

- Viewer creation failures include context identifying the viewer endpoint.
- A failed retry reports the final scoped request status.
- Pollers log failures and continue at the normal polling interval.
- No `unwrap()` is introduced in production paths.

## Tests

HTTP-level client tests use a local Axum server and real `DaemonClient`
requests to cover:

1. A scoped request with no cached token creates a viewer and succeeds.
2. A stale token receiving `404` creates a new viewer and retries once.
3. A non-authentication server error does not trigger viewer recreation.
4. Repeated `401`/`404` responses stop after one refresh and one retry.

Existing scoped isolation and TUI component tests remain unchanged and must
continue passing. Final verification runs formatting, Clippy with warnings
denied, and the full test suite.

## Security impact

The change does not weaken agent isolation. Viewer tokens remain generated by
the trusted daemon, stored only in process memory on the TUI side, and required
for every scoped view request. Refreshing a token grants the same UI viewer
capability as initial startup; it does not accept model-provided identities or
capabilities.
