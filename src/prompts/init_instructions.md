请为本仓库生成两份文档：**WGENTY.md**（项目事实文档）和 **AGENTS.md**（行为准则）。它们会在每个会话中被加载到系统提示中。

**重要：每个阶段需要实际操作工具（如 `task`、`file_write`、`ask_user_question` 等），不只是用文字描述计划。必须实际调用工具来完成任务。**

## 第一阶段：询问用户要设置什么

使用 AskUserQuestion 了解用户想要什么：

- **"你想设置哪些配置文件？"**
  选项："仅项目 WGENTY.md + AGENTS.md" | "加上个人 WGENTY.local.md" | "以上全部"
  描述：
  - 项目版 WGENTY.md：团队共享、纳入版本控制的项目事实文档——架构、命令、模块说明等。
  - AGENTS.md：团队共享的行为准则——代码风格、PR 规范、工作流约定。
  - 个人版 WGENTY.local.md：个人偏好（加入 .gitignore）——你的角色、沙箱 URL、工作流习惯。

- **"是否同时设置技能和钩子？"**
  选项："技能 + 钩子" | "仅技能" | "仅钩子" | "都不要，只要文档"
  描述：
  - 技能：按需调用的能力（`/技能名`）——重复性工作流和参考知识。
  - 钩子：工具事件上运行的确定性 shell 命令——AI 无法跳过。

## 第二阶段：全面探索代码库

使用 `task` 工具（`subagent_type` 设为 `"explore"`，`background` 设为 `true`，`use_small_model` 设为 `true`）并行启动多个子代理，彻底调查代码库。每个子代理负责一个维度，从以下所有方面收集信息。**必须实际调用 task 工具，不要只描述你要做什么。**

探索性任务用小模型更经济快速。如果小模型未配置（`small_model` 为空），会自动 fallback 到主模型。

后台子代理启动后立即返回 task_id，不会阻塞。所有子代理并行执行。在下一轮对话中，系统会自动注入已完成子代理的结果。如果结果尚未全部返回，使用 `task` 工具的同步模式（`background: false`）等待剩余结果。

## 第三阶段：补充信息缺口

使用 AskUserQuestion 询问代码无法回答的问题。

**针对可选项**：如果用户选择了个人版，询问角色、偏好等。

## 第四阶段：撰写 WGENTY.md（项目事实文档）

使用 `file_write` 工具在项目根目录编写 **WGENTY.md**。**重要：WGENTY.md 内容较多，必须分多次写入，每次只写 1-2 个章节（约 2000-4000 字符），避免单次 JSON 过长导致截断。**

**写入流程**：
1. 第一次 `file_write`：创建文件，写入文件头（`# WGENTY.md`...）和第一个章节（如"构建与运行命令"）
2. 后续使用 `file_edit` 工具追加章节：在文件末尾的 `<!-- APPEND -->` 标记处替换插入新章节
3. 每次 `file_edit` 替换 `<!-- APPEND -->` 为当前章内容 + 新的 `<!-- APPEND -->`
4. 最后一次 `file_edit`：替换 `<!-- APPEND -->` 为最后一个章节（不再保留标记）

文件前缀（第一次 `file_write` 时包含 `<!-- APPEND -->`）：

```markdown
# WGENTY.md

此文件为 Wgenty Code（claude.ai/code）在此仓库中工作时提供指导。
```

将第二阶段收集的所有**项目事实**组织到章节中。采用类似现有 `WGENTY.md` 的风格——`##` 标题、代码块、列表。

**包含**：
- 全部构建/测试/lint/运行命令及其用法
- 架构分层和模块间依赖关系
- 每个核心模块的职责和关键类型
- 配置项和环境变量（包括查找优先级）
- 所有 feature flags 和条件编译
- 关键依赖及其作用
- 已知的坑和历史架构变更
- CI/CD 流程
- API 端点（如有）

**不做**：
- 不写泛泛的开发建议
- 不编造不存在的信息
- 不过度重复 README（但转述核心项目事实）

**token 预算要求：WGENTY.md 总长度控制在 ~1500 tokens 以内（约 2000-3000 中文字符或 5000-6000 英文字符）。这会加载到每个会话中，越长越费钱。** 命令示例尽量单行、架构描述用简图不展开、目录树只列顶层。章节间用 `---` 分隔。

如果 WGENTY.md 已存在：读取并更新每个章节，补充新信息。

## 第五阶段：撰写 AGENTS.md（行为准则）

使用 `file_write` 工具，`path` 参数设为 `"AGENTS.md"`，在项目根目录编写**AGENTS.md**。AGENTS.md 一般较短，可一次性写入。如果内容超过 3000 字符，参照第四阶段分块写入。

文件格式——采用 `---` 分隔的多个章节（每个章节是一个独立的 section，会被分别加载）：

```markdown
# 章节标题

章节内容。
---
# 另一个章节

另一个内容。
```

**包含**：
- 代码风格约定（缩进、命名规则、注释规范等）
- 分支命名和 PR 规范
- Commit 信息风格
- 工作流约定（如"复杂变更先写计划"、"重构时解释权衡"）
- Review 注意事项

**不做**：
- 不重复 WGENTY.md 中已有的命令和架构信息
- 不写泛泛的建议

如果 AGENTS.md 已存在：读取它，提出具体的补充或修改，不静默覆盖。
